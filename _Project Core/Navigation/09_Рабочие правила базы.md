# 09_Рабочие правила базы

## Назначение

Этот документ фиксирует рабочие правила ведения базы Альгоны. Он нужен, чтобы при создании, редактировании и переносе лора сохранялись единый стиль, корректные связи, стабильная терминология и аккуратная работа с авторскими словами.

Правила из этого файла учитывать при правках навигационных документов и лорных заметок. Если правило из этого файла конфликтует с прямым указанием автора в текущем чате, следовать текущему указанию автора.

## Общие принципы работы с базой

- Не превращать навигационные документы в полноценные энциклопедические статьи.
- Не превращать лорные заметки вне папки Navigation в навигационные документы.
- В навигационных документах давать краткую сводку, опорные связи и направление к подробным файлам.
- Подробный лор выносить в профильные заметки или специальные навигационные документы.
- Не удалять авторские факты без прямого указания пользователя.
- При сокращении текста переносить важные сведения в подходящий файл, а не стирать.
- Не добавлять новые каноничные факты без пометки, если они являются догадкой, идеей или рабочим вариантом.
- Разделять канон, рабочие предположения, идеи и нерешённые вопросы.
- Существенные спорные места не решать самостоятельно: спросить автора в чате и обсудить.

## YAML / frontmatter

Для навигационных документов использовать стабильный YAML-формат:

```yaml
title: Название документа
id: nav-xx-short-id
type: meta
meta_kind: navigation_entry
status: in_progress
summary: "Краткое описание документа. Если в строке есть двоеточие, значение обязательно брать в кавычки."
tags:
  - projectcore
  - navigation
  - algona
created_at: YYYY-MM-DD HH:mm
updated_at: YYYY-MM-DD HH:mm
```

Правила:
- `summary` лучше всегда брать в кавычки.
- `tags` оформлять списком с отступом в два пробела.
- Не оставлять шаблонные плейсхолдеры вида `{ VALUE:title }`.
- Не добавлять `aliases` и `priority` в навигационные документы, если пользователь отдельно не решит иначе.
- `id` должен быть стабильным и не меняться без причины.

## Работа с авторскими именами, терминами и склонениями

Все авторские собственные имена, топонимы, этнонимы, названия рас, стран, миров, эпох, веществ и магических понятий считать русскоязычными словами, если пользователь явно не указал обратное.

Большинство авторских слов Альгоны подчиняется обычным правилам русского языка. Их нельзя переделывать «на слух» или подгонять под более привычные формы.

### Основной принцип

Не изменять основу авторского слова.

Склонять можно, но только от точной исходной формы, по ближайшей русской морфологической модели.

Запрещено:
- заменять буквы внутри основы;
- подгонять слово под более привычное написание;
- превращать `-ея` в `-ия`, `-ар` в `-ор`, `-хор` в `-хорд` и т.п.;
- исправлять авторское написание без прямого указания пользователя;
- склонять слово так, будто оно имеет другую начальную форму;
- смешивать похожие, но разные имена.

### Алгоритм склонения

1. Сначала определить точную начальную форму слова.
2. Не менять внутренние буквы основы.
3. Найти ближайшую русскую модель склонения.
4. Склонять только окончание, если русская модель это позволяет.
5. Если модель неясна — не угадывать, а использовать начальную форму в нейтральной конструкции.
6. После уточнения пользователя добавить слово в словарь склонений.

### Если нет уверенности

Явно уточнять у пользователя или использовать конструкцию без склонения:

- материк Эллагея;
- страна Энтория;
- народ Раан'Кхоры;
- мир Вирс'ахт;
- материк Дунтраст-Ахт.

Такой вариант временный. После уточнения склонения лучше использовать нормальную русскую форму.

### Зафиксированные нестандартные формы

- `Риер'Кахара` не склоняется во всех формах.

## Работа с идеями, догадками и каноном

При работе с лором различать:
- каноничные сведения;
- рабочие сведения;
- догадки;
- идеи;
- нерешённые вопросы.

Если информация не подтверждена пользователем, не оформлять её как окончательный канон.

Для пометок использовать короткие теги в тексте:
- `[ДОГАДКА]` — предположение на основе неполных данных.
- `[ВЫВОД]` — логический вывод из имеющихся данных.
- `[СОВЕТ]` — рекомендация к действию.
- `[НЕ УВЕРЕН]` — информация с существенной неопределённостью.
- `[ИДЕЯ]` — предложенный вариант развития лора, не являющийся каноном.

## Правила редактирования через GitHub

- По умолчанию ассистент работает в ветке `assistant`, а не напрямую в `main`.
- В `main` коммитить только по прямому указанию пользователя.
- По возможности делать правки отдельными понятными коммитами.
- Сообщение коммита должно кратко описывать изменение.
- Не смешивать в одном коммите несвязанные типы правок: например, исправление YAML и крупный перенос лора.
- При переносе сведений между файлами проверять, что информация не потерялась.
- После изменения файла сообщать название ветки, SHA коммита и список изменённых файлов.
- Если пользователь правит файл локально и не запушил изменения, сначала рекомендовать закоммитить локальные правки, затем делать `pull --rebase`.

## Что нельзя делать без прямого указания пользователя

- Массово переименовывать сущности.
- Менять каноничные названия рас, стран, миров и эпох.
- Удалять любые уже написанные фрагменты в любых заметках без уведомления об их удалении.
- Удалять большие блоки лора без переноса или явного разрешения.
- Исправлять авторские слова по собственному ощущению благозвучия.
- Добавлять крупные новые сущности мира как канон.
- Менять структуру папок базы.
- Переписывать тон мира в более мягкую, героическую или стандартно-фэнтезийную сторону.
